# Module 1: App Shell + Observability

**Complexity:** ⚠️ **Medium** — Touches Supabase Auth, FastAPI JWT verification, OpenAI Responses streaming semantics, and LangSmith wiring; expect iteration on API/SDK details as OpenAI ships minor reference updates.

**Single-pass feasibility:** An agent can complete the skeleton end-to-end in one focused session if Supabase and API keys are already available; first-time Supabase redirect URLs and OpenAI vector-store quotas often need a short second pass.

**PRD reference:** [Module 1: The App Shell + Observability](../PRD.md) — Auth, chat UI, OpenAI Responses API (threads via `previous_response_id` / stored conversation state + `file_search`), LangSmith tracing.

**Out of scope for Module 1 (defer to Module 2+):** Dedicated ingestion screen, pgvector, custom chunking pipeline, OpenRouter/Chat Completions, Supabase Realtime for ingestion status.

**Stack constraints (from CLAUDE.md):** Raw OpenAI Python SDK only (no LangChain/LangGraph); Pydantic for request/response bodies where helpful; stream chat via SSE; every Postgres table has RLS.

---

## Goals

1. Signed-in users only; Row-Level Security on all application tables.
2. Default UI is **Chat**: list conversations, open thread, send message, see assistant reply stream.
3. Backend proxies **Responses API** with **`stream=true`**, forwards SSE to the client, preserves continuity via **`previous_response_id`** (OpenAI-stored conversation state — “managed threads”).
4. **Managed file search:** documents uploaded for search live in **OpenAI vector stores** for Module 1 (not Supabase Storage yet unless you choose to mirror later); `file_search` tool uses `vector_store_ids`; optionally request `include: ["file_search_call.results"]` when you want citations surfaced.
5. **LangSmith** traces runs initiated from the backend (each user message / response generation visible in LangSmith).

## Task 2 — Supabase schema: conversations + OpenAI vector store metadata (RLS)

**Do:**

- SQL migration(s) under `supabase/migrations/` (or documented folder) creating:

  - `conversations` (minimum columns): `id` (uuid, PK), `user_id` (uuid, matches `auth.users.id`), `title` (text, nullable), `openai_previous_response_id` (text, nullable — last response id for continuity), `openai_vector_store_id` (text, nullable — optional if one store per conversation), `created_at`, `updated_at`.

  - Prefer **one vector store per conversation** OR **one per user**; pick one and document it. Per-conversation simplifies “what files apply to this thread.”

- Enable RLS on `conversations`; policies: `SELECT/INSERT/UPDATE/DELETE` only where `auth.uid() = user_id`.
- Use Supabase **service role** on the server for inserts/updates that must not be forgeable from the client, **or** restrict inserts to authenticated users via policies — either pattern is valid if every row is scoped by `user_id`.

**Validation:**

- Run migration against a dev project.
- In SQL editor as authenticated test user A: cannot read user B’s rows.
- Insert conversation row for user A succeeds when `user_id = auth.uid()`.

## Task 4 — Backend: verify Supabase JWT on protected routes

**Do:**

- Extract `Authorization: Bearer <access_token>` on protected routes.
- Validate JWT signature against Supabase JWKS (`SUPABASE_URL` / issuer claims per Supabase docs); reject expired/invalid with 401.
- Dependency injects `current_user_id: UUID` for handlers.

**Validation:**

- Request without token → 401.
- Request with forged/expired token → 401.
- Request with valid user token → 200 on a stub `GET /api/me` returning `{ "user_id": "..." }`.

## Task 5 — OpenAI: vector store lifecycle + file upload (server-side)

**Do:**

- Implement service module using **OpenAI Python SDK** (async or sync + thread pool — pick one style for the codebase).
- On conversation create (see Task 6): create vector store (if using per-conversation stores), persist id in `conversations.openai_vector_store_id`.
- Endpoint or internal helper: upload user file (pdf/text/md — start with formats OpenAI file search supports), attach file to vector store, poll/wait until processed enough for search per OpenAI docs.

**Security:**

- Enforce size/type limits; strip secrets from filenames; never expose `OPENAI_API_KEY` to the client.

**Validation:**

- Upload a small `.txt` fixture via API; OpenAI dashboard or API retrieval shows file attached to the correct vector store.
- Second user cannot target another user’s `conversation_id` (authorization check before any OpenAI mutation).

## Task 6 — Backend: Chat API — create conversation, send message, stream Responses SSE

**Do:**

- `POST /api/conversations` — creates DB row + vector store; returns conversation id.
- `GET /api/conversations` — lists current user’s conversations (optional pagination).
- `POST /api/conversations/{id}/messages` — body: `{ "text": string }`; loads conversation, reads `openai_previous_response_id`.

**OpenAI request shape (conceptual — align with current SDK):**

- Model from env (e.g. `OPENAI_RESPONSES_MODEL`).
- Tools include `{ "type": "file_search", "vector_store_ids": ["..."] }` when the conversation has a vector store.
- `stream=True`; iterate streamed events; translate to **your** SSE format for the frontend (e.g. `event: token`, `event: file_search`, `event: done`, `event: error`).
- On completion, persist new `openai_previous_response_id` from the response object **server-side** (single source of truth).

**Include:** Optionally pass `include=["file_search_call.results"]` so the UI can show citations.

**Validation:**

- Integration test or manual script: two sequential messages on same conversation without sending history from DB prove continuity via stored `previous_response_id`.
- Streaming: first byte arrives before full completion (observe timing or log deltas).

**Do:**

- Auth screens: sign-in / sign-up / magic link (match Supabase config).
- Protected route wrapper: unauthenticated users see auth; authenticated users see chat layout.
- Chat layout:

  - Sidebar or drawer: conversation list; “New chat”.
  - Main: message list + composer.
  - Calls backend with `Authorization` bearer token from Supabase session.

**SSE client:**

- Use `fetch` + `ReadableStream` or `EventSource` if compatible with auth headers (often `fetch` + manual SSE parse is easier with Bearer tokens).

**Validation:**

- Full manual flow: create account → new conversation → send message → streamed reply visible within 10s for short reply.
- Refresh page: conversation list still loads (RLS + correct user).

## Risk register

| Risk | Mitigation |
|------|------------|
| OpenAI SDK parameter names drift vs docs | Pin SDK version; link to official reference in code comment at call site. |
| SSE proxy bugs (buffering) | Disable gzip buffering for stream route; nginx doc if deployed later. |
| JWT verify subtlety (audience/issuer) | Follow Supabase “verify JWT” guide for your auth setup exactly. |
| LangSmith + async streaming | Use supported tracing pattern for generators in LangSmith docs. |
